## 🎯 Project Overview

This project demonstrates how a university student can apply **data automation techniques** — typically used in business intelligence and DevOps — to a deeply personal challenge: surviving and succeeding in Korea's highly competitive tech recruiting season (공채/수시 채용).

Rather than managing applications in a passive spreadsheet, this notebook turns raw application data into a **living, self-updating analytics system** that answers six critical questions every job-seeker should be asking:

| Question | Answered In |
|---|---|
| What is my current offer / interview / ghost rate? | Part 2 — Status Dashboard |
| Which applications need a follow-up right now? | Part 3 — Alert System |
| At which stage am I losing candidates? | Part 4 — Funnel Analysis |
| Which platform is generating the best ROI? | Part 5 — Platform Intelligence |
| Am I applying consistently, or in panicked bursts? | Part 6 — Timeline Analysis |
| Can I automate my weekly progress report? | Part 7 — Report Generator |

### Scope

- **400 simulated applications** spanning January–April 2024
- **30 Korean and global tech companies** (Naver, Kakao, Samsung SDS, Toss, Coupang, Krafton, LG CNS, SK Telecom, Google Korea, Amazon Korea, Microsoft Korea, and more)
- **Both Internship and Full-time** position types
- **All major Korean recruiting platforms** (Wanted, Jumpit, LinkedIn, Company Website, Saramin, Jobkorea)

## 🔧 Feature Engineering

Seven new columns are derived from raw data during Part 1. These features power every downstream analysis:

### `days_to_response`
```python
df["days_to_response"] = (df["response_date"] - df["date_applied"]).dt.days
```
Integer count of days between application submission and the company's first response.
`NaN` for ghosted/pending applications. Used in Part 6's response-time histogram.

### `salary_mid`
```python
df["salary_mid"] = (df["salary_min"] + df["salary_max"]) / 2
```
Midpoint of the advertised salary band. Using the midpoint rather than min or max gives a fairer comparison across companies with different band widths. Expressed in KRW; divided by 10,000 (만원) in visualisations.

### `month_applied`
```python
df["month_applied"] = df["date_applied"].dt.month_name()
```
Full English month name. Used as the row index in the Part 6 heatmap. Reindexed to `["January", "February", "March", "April"]` to enforce chronological order regardless of pandas' default alphabetical sort.

### `week_applied`
```python
df["week_applied"] = df["date_applied"].dt.isocalendar().week.astype(int)
```
ISO 8601 week number (1–53). Used as the x-axis in Part 6's weekly trend line chart.

### `days_since_applied`
```python
df["days_since_applied"] = (TODAY - df["date_applied"]).dt.days
```
Age of the application relative to the frozen `TODAY` constant (`2024-04-10`). Used in the Part 3 company-age horizontal bar chart.

### `got_response`
```python
df["got_response"] = df["response_date"].notna()
```
Boolean `True`/`False` flag. `True` if `response_date` is not `NaT`. Used as the denominator for response rate calculations across all segmented analyses.

### `reached_interview`
```python
df["reached_interview"] = df["status"].isin(["Interview", "Offer"])
```
Boolean flag indicating whether the application progressed past the screening stage. Both `Interview` and `Offer` statuses qualify, since an offer implies interviews were completed.

## 💡 Key Findings & Insights

The simulated dataset was constructed to reflect realistic job-search dynamics observed in Korea's tech recruiting market. The following patterns are built into the data and confirmed by the notebook's analyses:

### 1. Referrals have an outsized impact

Applications submitted with an employee referral (`referral = "Yes"`) are assigned higher probabilities for `Interview` and `Offer` statuses during data generation. The funnel analysis in Part 4 will consistently show:

- **Referral applications** → ~20–30% interview rate, ~15–20% offer rate
- **Cold applications** → ~15–22% interview rate, ~8–12% offer rate

**Takeaway:** Even a single warm introduction can double your offer probability.

### 2. Resume v1 underperforms v2 and v3

The dataset encodes a gradual improvement across resume versions:
- `v1` was used primarily in January, before the student iterated based on feedback
- `v2` reflects a mid-season refresh
- `v3` incorporates keyword optimisation and quantified achievements

Part 5's resume performance chart should show increasing interview and offer rates from v1 → v3.

### 3. Company Website applications have the highest response rate

Direct applications through a company's own portal bypass ATS black-hole algorithms that filter aggregator submissions. Part 5's platform intelligence table typically shows:

```
Company Website   ~58–65% response rate
Wanted            ~45–55% response rate
LinkedIn          ~40–52% response rate
Jumpit            ~38–50% response rate
```

### 4. Most rejections happen at resume screening

Part 4's rejection stage chart shows that the majority of rejections never reach a human reviewer. This is consistent with the reality of high-volume recruiting at large Korean conglomerates (대기업) where hundreds of applicants compete for each position.

**Takeaway:** ATS optimisation (keyword matching, clean formatting) has higher ROI than interview prep for students still below the interview conversion threshold.

### 5. Application volume peaks in mid-February (week 6–8)

Part 6's weekly timeline chart shows a clear spike in early February, corresponding to when Korea's spring recruitment cycle (상반기 공채) typically opens. Volume tapers off in March–April as most open requisitions close.

## 🎨 Design Decisions

### Why freeze `TODAY` instead of using `pd.Timestamp.today()`?

Reproducibility. A notebook that produces different numbers every time it runs is impossible to debug or grade fairly. By fixing `TODAY = pd.Timestamp("2024-04-10")`, every run — whether in a student's local Jupyter, a TA's Colab, or a CI pipeline — produces identical output.

For production deployment (real data, real scheduling), the constant is trivially replaced with `pd.Timestamp.today().normalize()`.

### Why `df_active` for follow-up alerts rather than filtering inline?

The alert system (Part 3) needs to be consulted multiple times: once for the printed digest, once for the bar chart, and once in the report generator (Part 7). Creating `df_active` as a named subset eliminates repeated filtering and ensures all three usages stay perfectly in sync.

### Why `STATUS_COLORS` as a dictionary rather than inline hex strings?

Consistency. With five status values appearing across eight charts, hardcoding colours inline would risk accidental inconsistencies (e.g., `Offer` appearing green in the donut but blue in a bar chart). A single shared dictionary guarantees every chart uses identical colours, making cross-chart comparisons intuitive for the reader.

### Why use `errors='coerce'` in `pd.to_datetime()`?

`response_date` is intentionally blank for ~40% of rows (ghosted and still-pending applications). Using `errors='raise'` (the default) would crash the notebook on the first empty string. `errors='coerce'` silently converts them to `NaT`, which pandas handles gracefully in all arithmetic and groupby operations.

### Why `isocalendar().week` instead of `.dt.week`?

`.dt.week` is deprecated as of pandas 1.1.0 and removed in later versions. `isocalendar()` returns a DataFrame with `year`, `week`, `day` columns following the ISO 8601 standard — more explicit and forward-compatible.

### Why 130 DPI for matplotlib figures?

`130` is a deliberate compromise between Colab's retina-capable display (which renders at 96 DPI by default and looks blurry for data-dense charts) and the overhead of `200+ DPI` which significantly slows rendering. At 130 DPI, labels on the heatmap remain legible without noticeably increasing cell execution time.
